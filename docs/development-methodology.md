# 開発手法

## 決定事項

- 開発は t-wada 流 TDD を基本とする。
- Red-Green-Refactor のサイクルを小さく回す。
- TODO リストを小さく保ち、仮実装、三角測量、明白な実装を状況に応じて使い分ける。
- Unit Test / Component Test は原則として先に書く。
- バグ修正では、まず再現テストを書いて失敗を確認してから修正する。
- 既存挙動を保護する characterization test は例外として許可する。
- Integration Test / E2E Test は実装後に主要導線を検証する目的で後追い作成してよい。
- Integration Test / E2E Test も作らない理由にはしない。重要な導線や外部連携は作成・実施する。
- ユーザー操作、状態分岐、フォーム、API 連携を持つ UI コンポーネントには Component Test を作成する。
- 純粋な静的表示だけのコンポーネントは、上位の Component Test / E2E Test で保護できる場合は個別テストを省略してよい。
- E2E Test は主要ユーザーフローに対して作成・実施する。
- 認証、課金、DB 書き込み、外部 API 連携を含む導線は優先して E2E Test で保護する。

## コメント方針

- テストには、意味のある塊ごとに日本語コメントで何のためのテストなのかを書く。
- 実装コードにも、仕様判断や非自明な処理の塊ごとに日本語コメントで意図を書く。
- テストコードにも、準備、実行、検証などの意味のある塊ごとに日本語コメントを添える。
- 自明な処理を説明するだけのコメントは避ける。

## 避けること

- ad-hoc な変更を避ける。
- 新規実装や仕様変更で、テストを後付けするだけの進め方を避ける。
- 根拠のないフォールバックや場当たり的な分岐を避ける。
- フォールバックを入れる場合は、仕様、外部 API の制約、ユーザー体験、安全性のいずれに基づくかをコメントやテストで明示する。
- 予期しない状態は可能な限り fail fast し、握りつぶさない。

## 実装方針

- 「シンプル」は最小コード量ではなく、目的に対して直線的で、読み手が理由を追える構造を指す。
- 場当たり的な分岐や局所的なパッチで逃げず、根本原因を解決する。
- 条件分岐は仕様、外部 API の制約、権限、環境差分など説明可能な理由がある場合に追加する。
- 条件分岐を追加した場合は、その分岐を通るテストを用意する。
- テストで意図が十分に表現できない場合は、日本語コメントで理由を補う。
- 抽象化は、責務を明確にし、重複や複雑さを減らす場合に行う。

## Lint / Format

- linter / formatter は Biome に統一する。
- Biome はデフォルト設定を基本とする。
- インデント、クォート、セミコロンなどの好みで設定を変えない。
- Biome が対応しない、または不十分なファイル種別のために別の linter / formatter を追加しない。
- Prettier、ESLint、markdownlint、stylelint などを、Biome の穴埋め目的で追加しない。
- 追加が必要な場合は、ユーザー確認を必須とする。

- Biome の具体的な導入タイミングと mise task への組み込み。
